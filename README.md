# 🎛️ Live MR Manager — Mod

> **[AutumnColor77/Live-MR-Manager](https://github.com/AutumnColor77/Live-MR-Manager)에서 갈라져 나온 개조(모드)판입니다.**
> 원본의 오디오 엔진·AI 분리·라이브러리·OBS 오버레이 위에, **노래 방송 스트리머**를 위한
> 기능(AI 가사 자동 정렬·통합 검색·MR 분리 가속·커스텀 모델)을 얹어 **독립적으로 릴리즈**합니다.

**Live MR Manager**는 실시간 방송·공연 환경을 위한 로컬 구동형 AI 음원 분리·오디오 제어
데스크톱 앱입니다. 이 모드는 그 위에서 **가사 싱크 자동화**와 **대량 곡 관리·처리**를
스트리머가 실전에서 쓸 수 있는 수준으로 끌어올리는 데 집중합니다.

| | 원본 (업스트림) | 이 모드 |
| :--- | :--- | :--- |

| **릴리즈** | 원 개발자가 관리 | **자체 릴리즈**(`v0.6.0-beta.x`)와 **자체 업데이트 알림** — 우리 버전끼리 이어짐 |
| **AI 가사 정렬** | — | **CTC 강제정렬 엔진**(한/영/랩혼합), 정렬 대기열, 3줄 가사, **정렬 전용 디리버브** |
| **오디오 출력** | 시스템 기본 장치 고정 | **실시간 장치 전환** + **모니터/MR 이중 채널**(장치별) + 2×3 라우팅 매트릭스 |

| **곡 추가** | 수동 입력 | **유튜브·가사 사이트 통합 검색**(공식 음원 우선 추천), **제목/가수 자동 추정** |
| **분류 체계** | 자유 입력 | **장르 / 카테고리 2축 표준화** + 기존 데이터 자동 재매핑 |
| **분리 대기열** | 세션 한정 | **껐다 켜도 복원**, 유튜브 실패 시 지수 백오프 재시도 |
| **MR 분리 속도** | RoFormer는 CPU로 곡당 20~30분 | **GPU 가속 팩(TensorRT)으로 곡당 ~30초** (아래 실측) |
| **커스텀 모델** | — | **가이드 + 원클릭 추천 다운로드 카탈로그** |
| **멜로밍 의존** | — | 멜로밍 없이도 기본 기능 완결(스트리머 우선, 범용 지향) |

핵심 오디오 엔진과 AI 분리, OBS 오버레이 등 **본체의 저작권은 모두 원 개발자**에게 있습니다.

## ✨ 이 모드에서 만들어진 것

### 🤖 AI 가사 자동 정렬
- **CTC 강제정렬 엔진**(Rust + ONNX Runtime): wav2vec2 모델로 가사 줄별 타임스탬프 자동 배치, 분리된 **보컬 스템 우선** 사용.
- **정렬 언어**: 한국어 / English / **랩·혼합(한+영)** — 랩 모드는 두 모델을 돌려 줄마다 우세 언어 채택.
- **정렬 전용 디리버브** — 리버브·이펙트가 심하면 정확도가 크게 떨어져서, 분리된 보컬의 잔향을 걷어낸 드라이 보컬로 정렬한다(정렬할 때만 실행·캐시, **재생엔 미적용**). GPU 가속 팩 필요, 없으면 원본 보컬로 폴백 → [docs/DEREVERB.md](docs/DEREVERB.md).
- **괄호 코러스 보존** — `사랑해 (사랑해)`처럼 실제로 불리는 코러스를 정렬 대상에 유지(곡 구조 지시어 `[Chorus]`만 제거). 예전엔 통째로 지워서 그 뒤 줄이 전부 밀렸다.
- 정렬 모델 인앱 다운로드(전용 Release), 정렬 대기열(복원 지원), 글자 필터 자동 정제.

### 🔎 노래 추가 통합 검색
- 유튜브 + 가사 사이트를 한 번에 검색, **공식 음원 우선 추천**("Provided to YouTube by …" / "Auto-generated by YouTube" 가중).
- 드래그·드롭 다중 추가, 곡 정보·장르·카테고리·일본어 3줄 여부까지 등록 시 지정.

### 🗂️ 장르 / 카테고리 표준화
- **장르**(사운드: 락·힙합·댄스…) 와 **카테고리**(씬/시장: K-POP·J-POP·애니…)를 분리한 단일 소스.
- 기존 라이브러리는 첫 실행 시 자동 재매핑. 기준 문서: [docs/GENRE_CATEGORY_STANDARD.md](docs/GENRE_CATEGORY_STANDARD.md).

### 💾 대기열 영속화 & 재시도
- MR 분리·AI 정렬 대기열을 저장·복원(앱/PC 재시작 후 이어서 처리).
- 유튜브 다운로드 실패 시 **지수 백오프 재시도**(30초 → 2분 → 5분 → 15분, 최대 4회).

### ⚡ MR 분리 GPU 가속 (이번 핵심)
- **선택 설치형 GPU 가속 팩(TensorRT)** — NVIDIA GPU에서 무거운 RoFormer 분리를 **곡당 ~30초**로. 없으면 기존대로 동작.
- 자세한 실측·원인 분석은 아래 [MR 분리 성능](#-mr-분리-성능--실측-분석), 구조는 [docs/GPU_ACCELERATION.md](docs/GPU_ACCELERATION.md).

### 🧩 커스텀 분리 모델
- `설정 → AI → 커스텀 모델`에 **가이드 + 추천 모델 원클릭 추가** 카탈로그. 프리셋(아키텍처)이 맞춰져 있어 실패하지 않음.
- 직접 받은 ONNX도 프리셋 지정 후 추가 가능(로컬 파일은 입력 형식 자동 검증). 가이드: [docs/CUSTOM_MODELS.md](docs/CUSTOM_MODELS.md).

### 🔊 오디오 엔진 (스트리머 실전용)
- **실시간 출력 장치 전환** — 재생 중에도 즉시 전환(위치·볼륨·피치/템포 유지), 선택 저장.
- **모니터 / MR 이중 채널** — 채널마다 다른 출력 장치(예: 모니터는 헤드폰, MR은 가상 케이블 → OBS). 채널×소스(보컬·인스트·메트로놈) **2×3 라우팅 매트릭스**로 각 채널에서 들을 소스를 고른다. MR 채널을 안 쓰면 기존 단일 출력과 100% 동일.
- **트랙 믹서** — 보컬/MR 독립 페이더 + 음소거·솔로.
- **메트로놈** — BPM 클릭(4박 강조), **탭 템포**(자동 탐지 BPM에서 시작해 두드려 보정), 재생 중 BPM 실시간 반영.
- **버스별 지연 보정** — 방송·녹음에서 목소리와 MR을 맞추기 위한 ms 오프셋(장치 버퍼 기반 추정값 제공).
- **출력 리미터** — 여러 소스를 합칠 때 생기는 클리핑 방지.

> 서로 다른 물리 장치로 나간 두 채널은 장치 클럭 차이로 미세하게 드리프트합니다(같은 장치로 묶인 소스끼리는 완벽 동기). 지연 보정은 고정 오프셋을 맞추는 것이지 드리프트를 없애지는 못합니다 — 다중 장치 오디오의 본질적 한계입니다.

### 📝 가사 싱크 편집기 & 오버레이
- 3줄(원문/차음/번역) 가사, 보컬 시작·간주 마커(수동+파형 자동+AI 첫 줄 기반 MV 인트로 감지), 인트로 자동 건너뛰기.
- 인앱 가사 호버창(크기·투명도 조절), OBS 오버레이 폰트 크기, 상단 키/BPM 표시.

## 📊 MR 분리 성능 — 실측 분석

RoFormer 계열(RawWaveform) 모델로 대량 분리 시 **곡당 20~30분, 58곡에 12시간+**가 걸리던
문제를 실측으로 추적해 잡았습니다. "ONNX라서 느리다"는 전제부터 틀렸습니다 — 같은 런타임에서
MDX/Kim Vocal 2 + DirectML은 곡당 18.5초였습니다.

### 원인 (3겹)
1. `ort` 빌드 피처에 `cuda`가 빠져 GPU EP 등록이 실패.
2. **ORT는 EP 등록 실패를 경고만 남기고 CPU로 계속 진행** → 로그엔 `Using: GPU (CUDA)`라 찍히며 실제로는 CPU로 도는 상태가 오래 은폐됨.
3. CUDA를 제대로 붙여도 CPU보다 느림 — 이 그래프는 **노드 5118개가 거의 융합되지 않아** 연산이 아니라 **커널 실행 오버헤드**에 묶임(GPU 사용률 7%, 41W/175W, 노드 실행 시간 합 5.0s vs 실제 19.3s, Gemm 4177ms×552 = 83%).

→ 런타임이 아니라 **그래프 융합**이 병목. 그래서 **TensorRT**(그래프 융합).

### 실측 (RTX 2070 8GB · Mel-Band RoFormer Deux · 입력 `[1,2,352800]` 정적 · 8초 청크)

| 경로 | 청크당 | 3분30초 곡 | 58곡 |
| :--- | ---: | ---: | ---: |
| 기존 앱 (CPU + GraphOpt Level1) | ~39,000 ms | 20~30분 | **12시간+** |
| CPU (Level3) | 14,113 ms | 8.2분 | ~8시간 |
| CUDA (cuDNN 존재) | 18,200 ms | 10.6분 | 더 느림 |
| DirectML | 크래시 `887A0006` (TDR) | — | — |
| **TensorRT (GPU 가속 팩)** | **~850 ms** | **~30초** | **~29분** |

- **정확도**: CPU vs TensorRT 출력이 소수점 8자리까지 동일(샘플 141만 개, mean=0.00027784 / rms=0.15003123). 품질 저하 없음.
- **엔진 캐시**: TensorRT 엔진은 GPU 아키텍처별(RTX 2070=`sm75`, 969MB)로 첫 빌드 ~150초, 이후 캐시 적중 시 로드 12~17초. GPU를 바꾸면 자동 재빌드.
- **함정**: ORT는 provider DLL을 제한된 검색 플래그로 열어 **PATH로는 의존 DLL을 못 찾음**(`Error 126`). Win32 `LoadLibraryExW`로 **절대경로 선적재**해 해결.

이 수정으로 GPU 팩이 없어도 **CPU 경로가 청크당 39초 → 14.1초**(Level3 + 정직한 폴백)로 빨라졌고,
NVIDIA GPU + 팩이 있으면 **25배 이상** 빨라집니다.

> ⚠️ GPU 가속 팩(cuDNN/TensorRT DLL ~2GB)은 NVIDIA 재배포 라이선스 검토가 필요해, 현재는
> 앱이 **상태 표시·폴더 열기**까지 제공하고 수동 설치를 안내합니다(자동 배포는 추후).

## 🏃 개발 환경

### 사전 요구
- **Windows 10/11** (릴리스·개발 타깃 Windows 전용)
- [Rust](https://www.rust-lang.org/tools/install) (rustup + stable)
- [Node.js](https://nodejs.org/) (LTS, v18+)
- [LLVM](https://releases.llvm.org/) / `winget install LLVM.LLVM` — `signalsmith-stretch` 빌드에 **libclang** 필요(`LIBCLANG_PATH` → `...\LLVM\bin`)
- [C++ Build Tools](https://visualstudio.microsoft.com/visual-cpp-build-tools/) — **「C++를 사용한 데스크톱 개발」** 워크로드

### 시작
```bash
npm install            # 의존성
npm run tauri dev      # 개발 모드
npx vitest run         # 프론트 테스트
cd src-tauri && cargo test --lib   # Rust 테스트
```

**Windows PowerShell 참고**
- `npm` 미인식 시 터미널 재시작 또는 `npm.cmd`
- 실행 정책 오류 시 `Set-ExecutionPolicy -Scope CurrentUser RemoteSigned`
- 캐시/데이터: `%LOCALAPPDATA%\com.autumncolor77.live-mr-manager\`, 관리형 도구: `%LOCALAPPDATA%\LiveMRManager\tools\`
